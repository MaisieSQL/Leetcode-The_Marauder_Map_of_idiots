---

## 🏗️ 模型定义 (Class)
类就是你在 **Bambu Studio** 里配置好的那个工程。它定义了：
* **能存什么数据**（耗材温度、层高）。
* **能执行什么操作**（切片、发送到打印机）。

当你点击“发送至拓竹 X1C”并完成打印，你就获得了一个 **Instance**。
* 它占据了物理空间（内存）。
* 它拥有了自己的属性（比如由于挤出机原因，它的颜色比模型定义的浅了一点）。

## ⚙️ 动态功能 (Method)
方法是模型自带的“技能包”。
* 调用 `my_model.auto_orient()` 会自动调整姿态。
* 这里的 `self` 就是在说：“调整**我这个**打印件的姿态，不要管打印板上其他的零件。”

---

## 💡 总结
* **Class** = 虚拟模型 (.stl/.3mf)
* **Instance** = 物理打印件 (Plastic part)
* **Attributes** = 打印参数 (Infill, Wall loops) -> 域(Field/Attribute)

# 🆔 深度解析：为什么 `self` 是实例的“身份证”？

> **核心逻辑**：`self` 解决了“我是谁”和“我要动谁的数据”这两个根本问题。

---

## 🔍 `self` 的三大职责

1. **指向性**：它告诉 Method（方法），现在正在处理的是内存中哪一个具体的“数据块”。
2. **连接性**：它把类里面的“功能”和实例里面的“数据”锁死在一起。没有 `self`，方法就找不到数据。
3. **独立性**：确保你在修改 `self.hp` 时，只扣除你自己的血，不会让屏幕上其他的怪莫名其妙死掉。

# 🏗️ 实例化审计：从“类文件夹”到“认主”的实例

> **核心逻辑**：界定对象（Self） -> 分配工具（Methods） -> 产出结果（Instance）。

---

## 📂 1. “类文件夹 (Class)”里装了什么？

在文件夹被打开之前，它里面装的仅仅是**“潜能”**：

* **逻辑格位 (Attributes)**：预留好的数据位置（比如：血量、坐标、账户名）。
* **动作模板 (Methods)**：定义的“怎么做”的脚本（比如：移动、扣款、解题）。
* **状态**：此时一切都是静止的，直到你下达“实例化”的指令。

---

## 🆔 2. “界定 Self”：给灵魂贴上身份证

当你执行 `Link = Hero()` 时，你是在下达一个生产指令：

1.  **内存开辟 (Allocation)**：在计算机内存里划出一块专属领地，这就是你的新“实例”。
2.  **绑定身份证 (Self Assignment)**：Python 自动把这块领地的“物理地址”发给 `self`。
3.  **分派工具包 (Method Binding)**：文件夹里的 Methods 瞬间识别到这个 `self` 身份证，并承诺：“从现在起，我的所有逻辑只为这个 `self` 身份证下的数据服务。”

---

## ⚡ 3. 调用的瞬间发生了什么？ (The Magic Moment)

编程最迷人的地方在于：你只需要界定**“它是谁”**，它就自动获得了**“它能干什么”**的所有能力。

* **你的操作**：界定 `self` 是“拓竹打印件 A”。
* **系统的反应**：自动将“模型文件夹”里所有的切片、控温、回抽逻辑（Methods）映射到“打印件 A”上。
* **审计结果**：你得到了一个不仅拥有独立数据，还自带**生存逻辑**的实体。

---

## 💡 这种设计的“审计优势”

* **界定清晰 (Boundary)**：以后调用 `Account_Audit_Folder` 时，只要界定 `self` 是“张三”，审计工具就会自动对准张三的流水，绝不越界。
* **互不干扰 (Isolation)**：张三的 `self` 身份证是唯一的，这意味着他的数据变动（Method 执行结果）绝不会污染到李四的数据块。
* **自动化 (Automation)**：一旦“界定”完成，实例就会自动跑出结果，无需你再手动干预内部复杂的逻辑流。

# ⚡ 深度审计：为什么 `__init__` 必须给初始值？

> **核心逻辑**：在数据分析中，我们习惯“先有数据，后有操作”；但在面向对象编程里，我们是“先造容器，并确保它出生即有数”。

---

## 1. 为什么要给初始值？（防止“空脑子”状态）

想象你去银行开户，柜员给了你卡，却没在系统里录入你的名字和身份证号：

* **后果**：这张卡就是一个无法识别的“空壳”。
* **程序崩溃**：当你后续调用 `get_customer()` 时，程序会因为在内存数据块里找不到任何信息而直接报 `AttributeError`。
* **结论**：`__init__` 的初始赋值是确保对象在“诞生”瞬间，就拥有了**生存所需的最小基础信息**。

---

## 2. 这个值有什么用？（后续行为的前提）

初始值不是摆设，它们是所有 **Methods（方法）** 能够跑通的**物理前提**。

在 `CreditCard` 类中，如果 `__init__` 没给初始值，后果如下：

| 缺失的初始值 | 导致的后果 | 逻辑死循环 |
| :--- | :--- | :--- |
| `_balance` (余额) | **还款动作** (`make_payment`) 无法执行 | 代码不知道该从哪个数字开始减法运算。 |
| `_limit` (额度) | **刷卡动作** (`charge`) 无法执行 | 代码无法判断 `0 + price` 是否超过了上限（上限是空的）。 |

> **金句**：初始值就像游戏里的“初始血量”。没有它，你一出生就 Game Over 了。

---

## 3. 变量名的秘密：`customer` vs `self._customer`

为什么在 `__init__` 里要写两次名字？这是初学者最容易绕晕的地方：

```python
def __init__(self, customer, bank):
    self._customer = customer  # 把“路人甲”存进“保险柜”
    self._bank = bank
```

> **审计逻辑**：没有初始录入（Initialization），就没有后续审计（Audit）。

---

## 🏗️ 1. 建立“基准线” (Baseline)
* **逻辑**：每个对象必须有一个明确的起始状态。
* **例子**：`_balance = 0`。如果不设为 0，当你第一次还款时，计算机会问：“从什么数字开始减？我不知道。”

## 📦 2. 数据的“持久化”存储
* **逻辑**：将外部传参转为内部属性。
* **类比**：
    - `customer` 是**瞬时流量**（你输入的名字）。
    - `self._customer` 是**库存余额**（存入内存的数据块）。
* **作用**：让对象在整个生命周期里都“记得”这些关键信息。

## ⚙️ 3. 为“行为 (Methods)”提供原料
* **逻辑**：所有的工具（Methods）都依赖属性（Attributes）来工作。
* **实战**：
    - `charge()` 需要读取 `self._limit` 才能判断是否超支。
    - 如果 `__init__` 没给 `_limit` 赋值，你的刷卡工具就是一块废铁。

# 🗺️ CreditCard 类定义 Python Code
```python
class CreditCard:
    """A consumer credit card."""

    def __init__(self, customer, bank, acnt, limit):
        """
        Create a new credit card instance.
        
        The initial balance is zero.
        
        customer: the name of the customer (e.g., 'John Bowman')
        bank:     the name of the bank (e.g., 'California Savings')
        acnt:     the account identifier (e.g., '5391 0375 9387 5309')
        limit:    credit limit measured in dollars
        """
        # 以下是【实例变量】：这些是属于 self（身份证）名下的私有数据块
        self._customer = customer
        self._bank = bank
        self._account = acnt
        self._limit = limit
        self._balance = 0  # 初始余额设为 0，为后续还款和刷卡逻辑建立“基准线”

    # --- 以下是不同的 Methods（行为/小工具包） ---

    def get_customer(self):
        """Return name of the customer."""
        return self._customer

    def get_bank(self):
        """Return the bank's name."""
        return self._bank

    def get_account(self):
        """Return the card identifying number (typically stored as a string)."""
        return self._account

    def get_limit(self):
        """Return current credit limit."""
        return self._limit

    def get_balance(self):
        """Return current balance."""
        return self._balance

    def charge(self, price):
        """
        Charge given price to the card, assuming sufficient credit limit.
        Return True if charge was processed; False if charge was denied.
        """
        # 逻辑审计：如果 现有债务 + 新消费 > 信用上限，则拒绝
        if price + self._balance > self._limit:
            return False
        else:
            self._balance += price
            return True

    def make_payment(self, amount):
        """Process customer payment that reduces balance."""
        self._balance -= amount
```

# 🛡️ 为什么需要 `if __name__ == "__main__":`？

> **核心逻辑**：区分“独立运行”与“被动导入”，防止测试代码污染其他项目。

---

## 🏗️ 逻辑对比

### ❌ 如果不加这行：
* 你在 `A.py` 里写了测试。
* 你在 `B.py` 里执行 `import A`。
* **后果**：当你运行 `B.py` 时，`A.py` 里的测试逻辑会自动跳出来乱跑，导致输出一堆你不需要的账单信息。

### ✅ 加上这行后：
* **场景 A (直接运行本文件)**：`__name__` 等于 `'__main__'`。
  - 结果：`CreditCard` 类被定义，且**测试代码运行**。
* **场景 B (被其他文件导入)**：`__name__` 不等于 `'__main__'`。
  - 结果：`CreditCard` 类被定义，但**测试代码保持沉默**。

---

# 🕵️‍♂️ 揭秘：'__main__' 到底是谁定义的？

> **核心真相**：它是 Python 解释器内置的一个“保留字”，代表着**“当前正在直接运行的脚本”**。

---

## 🔍 逻辑链条

1. **内置属性**：`__name__` 是 Python 给每个 `.py` 文件自动分配的身份证槽位。
2. **主角身份**：当你直接运行某个文件，Python 会自动把这个槽位填上 `'__main__'` 这个值。
3. **配角身份**：当这个文件被别人 `import` 时，Python 会把槽位填上它的**文件名**。

## 💡 审计类比
这就像是你去参加一个会议：
* 如果你是**主讲人**，你的工牌上写着 **“主角 (Main)”**。
* 如果你是**被邀请的顾问**，你的工牌上写着你的 **“名字 (CreditCard)”**。

`if __name__ == "__main__":` 就像是在入场处做一个检查：“只有工牌上写着‘主角’的人，才有权在这里大声做自我介绍（跑测试逻辑）。”

# 🎭 揭秘：Python 的“系统秘密档案” (__name__)

> **核心逻辑**：Python 会根据文件的“运行场景”自动分配身份标签。

---

## 1. 什么是 `__name__`？
在 Python 中，每个脚本文件诞生时都自带一个隐藏的“秘密档案”，档案的名字就叫 `__name__`。

* **主角身份**：当你**直接运行**一个 Python 文件时（比如点击运行按钮），Python 解释器会认为这个文件是当前任务的“主角”。
* **系统操作**：为了标记主角，Python 会在后台偷偷执行：`__name__ = '__main__'`。

---

## 2. 为什么平时看不见它？
`__name__` 就像电脑的后台系统进程，平时保持隐身。只有当你写下 `if __name__ == '__main__':` 时，你才是在调取这份档案进行**“身份比对”**。

* **你的提问**：“档案里的名字是 `'__main__'` 吗？”
* **Python 的审计结果**：
    * 如果是直接运行，档案里存的就是 `'__main__'`。
    * 如果是被别人 `import` 进来的，档案里存的就是它原本的文件名。

---

## 3. 身份比对规则表

通过下表可以清晰看出，你不需要手动赋值，Python 会自动执行以下规则：

| 运行场景 | `__name__` 档案里的值 | 是否等于 `'__main__'`？ | 结果 |
| :--- | :--- | :--- | :--- |
| **你直接双击运行本文件** | **`'__main__'`** | **YES** | **测试代码跑起来** ✅ |
| **你在别的脚本里 `import` 它** | `'CreditCard'` (文件名) | **NO** | **测试代码保持沉默** 🔇 |

---

## 💡 凌影的“身份审计”总结

* **'__main__' 是谁？** 它是 Python 内置的一个“主角标签”。
* **意义何在？** 这种设计确保了我的 `CreditCard` 类文件夹既可以作为独立程序进行自检（跑测试），也可以作为干净的工具包被别人借用，而不会带着一堆账单测试乱跑。

# 🐍 Python OOP 核心逻辑审计：以 Vector 类为例

在 Python 的面向对象编程（OOP）中，理解“谁在后台干活”以及“如何对接系统协议”是进阶的关键。以下是关于 **隐式调用**、**运算符重载** 与 **自定义方法** 的深度审计总结。

---

## 1. 核心定义与逻辑映射

| 概念 | 逻辑本质 | 语法表现 | 物理类比 (3D 打印 & 五金) |
| :--- | :--- | :--- | :--- |
| **隐式调用 (Implicit Call)** | **后台自动触发**。由 Python 解释器在特定语法下自动执行，无需手动点名。 | 所有的 `__xxx__` 方法 | **感应式零件**：只要你触发了特定感应器（如调用 `len()`），它就自动运行。 |
| **运算符重载 (Operator Overloading)** | **旧瓶装新酒**。借用 Python 预设的符号，换上自定义的内核逻辑。 | `+`, `-`, `*`, `//`, `[]` 等符号 | **标准双头螺栓**：外接口是标准的 M8 规格，但内部连接的是你自定义的支架。 |
| **自定义方法 (Custom Methods)** | **手搓新功能**。由开发者完全自主命名和定义的方法。 | `v.normalize()`, `v.deposit()` | **异形连接件**：没有标准螺口，必须按照你设计的独特名字来手动操作。 |

> **⚠️ 逻辑审计结论**：**运算符重载是隐式调用的一个子集。** 所有的运算符重载都是隐式触发的，但隐式调用（如 `__init__`）不一定涉及运算符。

---

## 2. 实例解构：Vector 类

这个 `Vector` 类通过定义一系列“双头螺丝”（Dunder Methods），完美适配了 Python 的原生协议。

```python
class Vector:
    """代表多维空间的一个向量。"""

    def __init__(self, d):
        """
        【隐式调用】：初始化协议。
        触发动作：v = Vector(5)
        注：_coords 是单下划线开头的“内部支架”，不建议外部直接访问。
        """
        self._coords = [0] * d

    def __len__(self):
        """
        【隐式调用】：对接系统 len() 函数。
        触发动作：len(v)
        """
        return len(self._coords)

    def __getitem__(self, j):
        """
        【运算符重载】：重载索引读取符号 []。
        触发动作：val = v[j]
        """
        return self._coords[j]

    def __setitem__(self, j, val):
        """
        【运算符重载】：重载索引赋值符号 []=。
        触发动作：v[j] = 10
        """
        self._coords[j] = val

    def __add__(self, other):
        """
        【运算符重载】：重载加号运算符 +。
        触发动作：v3 = v1 + v2
        逻辑：self 是左操作数，other 是右操作数。
        """
        if len(self) != len(other):
            raise ValueError('维度必须匹配')
        result = Vector(len(self))
        for j in range(len(self)):
            result[j] = self[j] + other[j]
        return result

    def __str__(self):
        """
        【隐式调用】：对接 print() 和 str() 协议。
        触发动作：print(v)
        """
        return '<' + str(self._coords)[1:-1] + '>'

    # --- 非运算符重载（自定义方法）示例 ---
    def normalize(self):
        """
        【自定义方法】：不对应任何符号，必须显式调用。
        调用方式：v.normalize()
        """
        pass
```
### 3. 快速审计技巧

* **识别隐式/重载**：认准 `__xxx__`（Dunder 方法）。它们是写给 **Python 解释器** 看的，目的是让你的类看起来像“原生数据类型”。
* **理解单/双下划线**：
    * `__add__`（双头螺丝）：一头连着你的代码，一头连着 Python 系统协议。
    * `_coords`（单头支架）：只在类内部使用的私有属性，不需要右侧下划线，因为它不与外部协议对接。
* **一句话总结**：
    * **重载** = 借用 Python 的 **名额**（符号），换上自己的 **内核**。
    * **自定义** = 放弃 Python 的 **符号**，打造自己的 **名号**。

## 1. LC 706. 设计哈希映射 (Design HashMap)
**难度：简单 (逻辑进阶)**

### 🔍 逻辑审计
这道题要求你手搓一个简化版的 Python `dict`。它能让你理解 Python 字典在底层是如何通过“哈希协议”找到位置的。

* **核心任务（接口对接）**：
    * `put(key, value)`：自定义方法，逻辑上等同于重载 `__setitem__`。
    * `get(key)`：自定义方法，逻辑上等同于重载 `__getitem__`。
    * `remove(key)`：自定义方法，逻辑上等同于重载 `__delitem__`。
* **为什么必刷**：你会接触到 **哈希函数 (Hash Function)** 和 **冲突处理 (Collision Handling)**，理解 `self._map` 这个“内部支架”是如何被划分成一个个“桶（Buckets）”的。

## 2. LC 146. LRU 缓存 (LRU Cache)
**难度：中等 (大厂高频)**

### 🔍 逻辑审计
这是一个经典的 **“非运算符重载”** 题目。它不使用 `+` 或 `[]`，而是完全依赖自定义方法来管理复杂的内部状态。

* **核心任务（复杂状态管理）**：
    * **`self._capacity`**：在 `__init__` 中定义的内存阈值。
    * **`self._cache` (Hash Table)**：实现 $O(1)$ 时间复杂度的查找。
    * **`self._usage_order` (Double Linked List)**：记录使用顺序。
* **为什么必刷**：这道题能极大地锻炼你对 **`self` 内部多重零件协同工作** 的掌控力。你需要确保在 `get` 和 `put` 的过程中，多个内部支架（哈希表和链表）的数据始终保持同步。

## 3. LC 303. 区域和检索 (Range Sum Query)
**难度：简单 (前缀和)**

### 🔍 逻辑审计
这道题是考察 **`__init__` 预处理能力** 的教科书级案例。

* **核心任务（生产线首站优化）**：
    * **不要在调用时计算**：如果你在 `sumRange` 里写 `for` 循环，系统会超时。
    * **在 `__init__` 里做足功课**：在对象出生时，就通过一次遍历计算出“前缀和”并存入 `self._preSum`。
* **为什么必刷**：你会深刻体会到 **“空间换时间”** 的架构思想。`__init__` 不只是初始化变量，更是为整个类的后续运行铺设“高速公路”。

## 💡 刷题总结 (Audit Tips)

1. **`__init__` 是关键**：始终思考哪些数据是可以预先计算并存入 `self._internal_data` 的。
2. **保护内部支架**：所有的存储变量建议使用单下划线开头（如 `self._storage`），提醒自己不要在类外部随意篡改。
3. **接口契约**：看清题目要求的是“符号调用”还是“方法调用”，合理选择 **隐式魔法方法** 或 **显式自定义方法**。

### 🚀 LeetCode 类设计刷题逻辑

1. **Focus on `self`**: 始终思考哪些变量需要作为“内部支架”存入 `self._xxx`。
2. **Init for Performance**: 能在 `__init__` 里算好的，就别留到方法调用时再算。
3. **Internal vs External**: 
   * 如果题目要求支持 `obj[i]`，记得用 `__getitem__`。
   * 如果题目要求 `obj.put()`，这是自定义方法。
  
# 🛠️ Python 类设计决策模型：运算符重载 vs. 自定义方法

在工程实践中，决定一个功能应该是“隐式符号触发”还是“显式方法调用”，是衡量一个开发者架构思维的关键。以下是核心审计逻辑与决策路径。

---

## 1. 逻辑审计决策树 (Decision Tree)

当你需要为一个类添加新功能时，请遵循以下审计流程：

1.  **第一步：需求直觉审计**
    * 该操作是否符合人类的“数学本能”或“容器直觉”？
    * *示例*：加减乘除、比大小、通过索引取值、打印对象。
2.  **第二步：协议清单审计 (The Home Depot Check)**
    * **IF (Python 官方有预设接口)**：必须使用对应的 `__xxx__` 魔法方法。
        * **结论**：执行 **运算符重载 (Operator Overloading)**。
    * **ELSE (无预设接口)**：你无法发明新符号（如 `@@` 或 `->`）。
        * **结论**：执行 **自定义方法 (Custom Methods)**。

---

## 2. 核心“双头螺丝”清单 (常用 Dunder Methods)

无需背诵全部，只需掌握以下高频标准件：

| 想要支持的操作 (扳手) | 必须对接的型号 (双头螺丝) | 逻辑类别 |
| :--- | :--- | :--- |
| `obj + other` | `__add__` | 算术运算 (Arithmetic) |
| `obj[i]` (读取) | `__getitem__` | 容器协议 (Container) |
| `obj[i] = x` (赋值) | `__setitem__` | 容器协议 (Container) |
| `len(obj)` | `__len__` | 容器协议 (Container) |
| `obj1 == obj2` | `__eq__` | 比较运算 (Comparison) |
| `print(obj)` | `__str__` | 表现协议 (Representation) |
| `v = Class()` | `__init__` | 生命周期 (Lifecycle) |

---

## 3. 非标准需求的解决方案 (自定义方法)

如果逻辑不在官方清单里（例如：给向量做“旋转” `rotate` 或“转账” `transfer`），必须走“自定义方法”路线：

* **动作**：手搓一个语义明确的名字，例如 `def rotate(self, angle):`。
* **代价**：无法使用简写符号，必须使用点语法：`v1.rotate(90)`。
* **好处**：**语义透明**。调用者一眼就能看出代码的业务意图，避免了过度使用运算符导致的歧义。

---

## ⚖️ 凌影的工程架构总结

### 🔹 运算符重载 (Operator Overloading)
* **场景**：高度符合数学模型或标准数据结构（List/Dict）的直觉。
* **原则**：严格匹配 Python 预设名 `__xxx__`。
* **优势**：语法简洁优雅 (`a + b`)，完美融入 Python 原生生态。

### 🔹 自定义方法 (Non-Operator / Custom Methods)
* **场景**：处理具体的业务逻辑、复杂动作或非标准运算。
* **原则**：自主命名，遵循 `lowercase_with_underscores` 规范。
* **优势**：灵活性极高，不受符号数量限制，代码自解释性强。
